Co uplisting na NASDAQ reálně znamená
Pozitivní efekty:

Větší likvidita — Na NASDAQ obchoduje mnohem více investorů, užší spread (rozdíl bid/ask)
Institucionální investoři — Mnoho fondů a ETF nemůže kupovat OTC akcie. Po uplistingu se otevírá obrovský pool nových kupců
Větší viditelnost — Pokrytí analytiky, zařazení do screenerů, mediální pozornost
Důvěryhodnost — NASDAQ má přísnější požadavky na reporting, governance, minimální cenu akcií
Rizika a realita:

"Buy the rumor, sell the news" — Cena často roste v očekávání uplistingu, ale po samotném uplistingu může klesnout, protože spekulativní zisk je už "zapečený" v ceně
Přísnější požadavky — Minimální cena akcie $4, audit financí, nezávislí členové board — pokud firma nesplní, hrozí delisting zpět
Diluce — Firmy často potřebují další fundraising k splnění požadavků NASDAQ (to už HGRAF udělal s tím C$30M)
Není záruka úspěchu — Uplisting sám o sobě nezmění fundamenty firmy
Typický vzorec u small-cap uplistingů
Fáze 1: Očekávání    → cena roste (spekulace, FOMO)
Fáze 2: Samotný den   → často peak
Fáze 3: Po uplistingu → může následovat výprodej (profit-taking)
Fáze 4: Dlouhodobě   → záleží čistě na fundamentech

Co se týče dnešního růstu +18% — to zapadá do toho vzorce fáze 1. Kombinace čerstvého kapitálu (C$30M), regulačních povolení a očekávání uplistingu vytváří silný momentum. Otázka je, kolik z toho je fundamentálně podloženo a kolik je spekulativní hype kolem grafénového sektoru.

Pro tento use case bych zvážil 3 přístupy:

1. Dlouhoběžící daemon (doporučuji)
Protože ib_async podporuje streaming headlines v reálném čase — nemusíte pollovat. Skript se připojí ke Gateway, subscribne se na HGRAF/BSWGF news a reaguje okamžitě, jakmile přijde headline.

systemd service → Python daemon → streaming z IB Gateway → Claude → Telegram

Nejnižší latence (zpráva → notifikace během sekund)
Žádný scheduler potřeba
systemd se postará o restart při pádu
Minimální nároky na VPS
2. Cron
cron (každých 15 min) → Python skript → reqHistoricalNews → Claude → Telegram

Jednoduché, spolehlivé, debugovatelné
Zpoždění až 15 minut (nebo kolik nastavíte)
Každý běh = nové připojení ke Gateway
Pro váš účel naprosto dostačující
3. Windmill
Overkill pro jeden agent/pipeline
Vyžaduje Docker + PostgreSQL → víc RAM na VPS
Dává smysl až když budete mít 5+ různých workflowů
Hezké UI pro monitoring, ale zbytečná komplexita teď
Verdikt
Daemon	Cron	Windmill
Latence	Sekundy	Minuty	Minuty
Složitost	Střední	Nízká	Vysoká
RAM navíc	~50 MB	0	~1 GB
Monitoring	journalctl	logy	Web UI
Škálovatelnost	Střední	Nízká	Vysoká
Pro start: cron. Jednoduché, funguje, snadno debugovatelné. Pokud zjistíte, že potřebujete okamžitou reakci na zprávy, přejdete na daemon. Windmill až pokud budete mít víc agentů.

Co to znamená
Pro news přes API byste musel přidat některé z těchto:

Provider	Cena/měsíc	Co dostanete
Briefing.com (BRFG)	Zdarma	Základní tržní komentáře
Benzinga Pro (BZ)	~$5	Kvalitní news, dobrý pro small-cap
Dow Jones Real-Time (DJ-RT)	~$25+	Premium zprávy
Fly on the Wall (FLY)	~$25	Trading-oriented news
Ale pozor — HGRAF je OTC small-cap. I placené news providery u IBKR budou mít pro tenhle ticker velmi omezenou pokrytí. Benzinga bývá na small-cap nejlepší z nich.

Alternativní přístup (možná lepší)
Pro small-cap grafénové akcie by mohlo být efektivnější:

Přímo z webu firmy — blackswangraphene.com/news, hydrograph.com/media
GlobeNewsWire / NewsFile — kde firmy publikují press releases
StockTitan — stocktitan.net/news/HGRAF
Yahoo Finance RSS — pro daný ticker
SEC/SEDAR filings — regulatorní podání
Tyto zdroje jsou zdarma a pro small-cap typicky obsahují víc relevantních zpráv než IBKR news feed. Nepotřebujete k tomu Gateway — stačí web scraping nebo RSS.

Revidovaná architektura
cron → Python skript → scrape news z webu/RSS → Claude API → Telegram
                       (žádný IB Gateway potřeba)

Tím odpadá celá složitost s IB Gateway + VPS. Můžete to spustit kdekoliv — i na levném VPS za $3/měsíc, nebo dokonce lokálně.
